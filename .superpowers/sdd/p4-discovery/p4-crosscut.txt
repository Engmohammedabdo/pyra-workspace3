Despite CLAUDE.md/MEMORY.md implying Phase 2 finished this file, it has ZERO i18n imports and 43 Arabic-containing lines. The i18n-check manifest never actually claimed this bare file (only MyWorkInbox.tsx, a sub-component it renders) — so the manifest is technically consistent, but the PLAN must explicitly decide whether Phase 4 absorbs this page's finance-labeled activity map (since the strings are finance-domain) or defers it as a named cross-phase carry-over (e.g. "Phase 4.5" or explicitly flagged for whichever phase eventually claims the dashboard-root admin shell). Silently ignoring it will leave a real, larger Arabic surface undetected by i18n:check since the manifest doesn't scan it.

2. **Contract-status wording collision between CRM and Finance surfaces for the SAME status value `in_progress`.** CRM's existing shadow map (documented in Phase-3 census, contract-card.tsx) uses "جاري"; the Finance admin surface's two independent shadow maps (contracts-client.tsx, purchase-orders-client.tsx's copy-pasted contract vocabulary) use "قيد التنفيذ". Phase 4 cannot silently pick one when building/reusing the `crmContract` catalog key for the finance UI — this is a live production wording inconsistency that predates the i18n effort and will become VISIBLE (both AR strings can't coexist under one catalog key) the moment Finance is wired to the same catalog entity CRM uses. Needs an explicit decision from Abdou: which word wins, or do they stay as two distinct catalog entities (finance vs crm) despite representing identical underlying data.

3. **Purchase-orders list-page STATUS_MAP is the wrong entity (copy-paste bug, pre-existing).** `app/dashboard/finance/purchase-orders/purchase-orders-client.tsx`'s STATUS_MAP + status-filter `<Select>` options use CONTRACT status vocabulary (draft/active/in_progress/completed/cancelled) instead of PO status vocabulary (draft/sent/acknowledged/received/invoiced/cancelled — the correct set is used one level down on the PO detail page instead, with 3 additionally-divergent wordings). This means the PO list's status filter dropdown is currently filtering on values that likely never match real `pyra_purchase_orders.status` rows (a functional bug, not just a translation gap) — swapping this file to `useStatusLabels('po')` during Phase 4 will surface (and effectively fix) this bug as a side effect. The plan should flag this explicitly rather than let an implementer "quietly" change behavior while doing what looks like a pure translation swap — get a decision on whether fixing this filter's correctness is in scope for Phase 4 or should be a separate bug-fix commit.
