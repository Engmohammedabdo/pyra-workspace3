7. HAZARDS section (business logic branching on Arabic strings; status display-vs-comparison risk)
================================================================

H1. [CRITICAL — data hazard] credit-notes/new/page.tsx L54: the auto-populated `reason` field default value is an Arabic template literal (`` `مرتجع / تعديل على الفاتورة ${inv.invoice_number}` ``) that gets PERSISTED VERBATIM to the DB as the credit note's `reason` column if the admin doesn't edit it before submit. This is displayed AND stored — translating the UI template literal changes what future readers of that credit note (including PDF/reports, if any) see as the stored reason, retroactively for NEW records created under an EN-locale session. Plan must decide: (a) keep this auto-fill text canonically Arabic regardless of UI locale (business record convention), or (b) let it follow UI locale and accept that credit note reasons will be bilingual/inconsistent going forward. Recommend (a) — do NOT make this ICU-translated without an explicit decision.

H2. [MEDIUM — numbering-system hazard, not a string] components/finance/contract-items-editor.tsx L124: `arabicLetters` ordinal scheme (أ ب ج د هـ و ز ح ط ي) for contract sub-item numbering. This is CONTENT that appears inside the generated Scope-of-Work section of contracts, which likely gets pulled into invoices/portal display (per the component's own EmptyState hint: "أضف بنود نطاق العمل لتظهر في الفواتير والبوابة" = "add scope items to appear in invoices and the portal"). If contract items are rendered to an English-locale client in the portal, showing Arabic ordinal letters (أ. ب. ج.) would look broken/foreign. This needs an explicit numbering-scheme decision (Arabic letters vs Latin letters vs roman numerals) tied to the CONSUMING locale (client's `preferred_language`), not just the admin dashboard's locale — cross-references the portal-parity concern from CLAUDE.md's "STOP — Ask WHO?" checklist.
