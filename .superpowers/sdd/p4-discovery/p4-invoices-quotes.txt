HAZARDS
================================================================
1. QuoteBuilder.tsx L576: `{t.name_ar || t.name}` — template display prefers the Arabic-named field (`name_ar`) over the base `name` field. This is a data-shape/business-logic decision baked into JSX, not a simple string literal — assumes Arabic-first UI. When EN locale ships, this needs either a locale-aware field choice (`locale === 'ar' ? t.name_ar : t.name`) or a schema decision (same class as p3's `stage.name_ar` DB-data flag). NOT a simple string swap.
2. quotes-analytics-client.tsx L52-62 STATUS_LABELS: independent hardcoded map with a live DRIFT BUG — `signed: 'موقع'` (missing shadda) vs the canonical `quote.signed: 'موقّع'` in messages/ar/statuses.json. This is a pre-existing data-quality bug that migration will silently fix IF the plan routes this file through useStatusLabels('quote') — flag explicitly so the fix is INTENTIONAL and documented, not an accidental side-effect buried in a large diff.
3. `.localeCompare(String(x), 'ar')` in both invoices-client.tsx and quotes-client.tsx sort comparators — NOT a business-logic branch on Arabic content per se, but a HARDCODED COLLATION LOCALE baked into client-side sort ordering. Under an EN locale, sorting Latin-script client/company names with Arabic collation rules could produce a different (arguably wrong) order than 'en' collation would. This is the same class of issue as p3's pipeline-client.tsx L174 finding, but doubled here (2 files, 2 lines each = 4 total sites) — the plan MUST parametrize these on the active locale, not just swap strings.
4. No `.includes('عربي')`-style checks, no Arabic string-equality branches, and no `window.confirm`/`alert` calls found anywhere in this file set (SignaturePad, both hooks, and the 12 UI files were all checked). The window.prompt() at QuoteBuilder.tsx L467 is UI-structural, not a business-logic-on-Arabic-string hazard, but is still flagged above under cross-cutting concerns since it can't be swapped via simple string replacement.
5. CURRENCIES const (QuoteBuilder.tsx L122-128) mixes AR and EN in a single joined label string per currency (`'AED — درهم إماراتي'`) — this is effectively a bilingual label baked into ONE string per option, meaning the EN catalog can't just be "the same key translated" — it needs a structurally different composed string (e.g. `'AED — UAE Dirham'`). Not a broken hazard today, but a migration-shape hazard: naive key-for-key translation would produce garbled mixed strings if done carelessly (e.g. translating only half the em-dash-joined string).
