16. Recharts-specific hardcoded labels (chart tooltip/legend/axis)
================================================================
ALL THREE chart components (RevenueExpenseChart, ExpenseBarChart,
ExpenseCategoryPieChart) hardcode Arabic strings INSIDE Recharts
Tooltip/Legend/Bar-label `formatter` callback functions — these execute as
plain JS closures, NOT JSX, so:
1. They will NOT be caught by a naive "scan JSX for Arabic" migration tool.
2. They CANNOT simply call `t()` unless the translation function is captured
   in the component's closure (i.e., call `useTranslations()` at the top of
   the component and reference `t('...')` inside the formatter callback —
   straightforward since these are function components, but must not be
   forgotten during the swap).
3. The "المبلغ" (amount) label appears independently in BOTH ExpenseBarChart
   and ExpenseCategoryPieChart Tooltip formatters (2 separate hardcodes of the
   same word) plus RevenueExpenseChart hardcodes 'الإيرادات'/'المصاريف' TWICE
   each (once in Tooltip formatter, once in Legend formatter) — 6 total
   hardcoded-in-closure Arabic string sites across 3 files needing coordinated
   translation-hook threading.
4. All three ALSO hardcode `toLocaleString('en-AE')` + literal `' AED'` suffix
   — per CLAUDE.md doctrine ("Currency/number formatting is locale-stable
   (en-AE) by design — do not localize digits in money") the en-AE part may be
   INTENTIONALLY correct, but the literal `'AED'` text should probably become
   a currency-code variable/prop rather than a hardcoded string, since these
   chart components have no currency prop today (implicitly assume AED site-
   wide, which is consistent with current all-AED production data per other
   locked decisions, but is a latent multi-currency gap like the CRM dashboard
   ai-insights AED assumption documented elsewhere in the codebase).

================================================================
17. HAZARDS section
================================================================
H1. app/dashboard/finance/page.tsx L30 — `toLocaleDateString('ar-EG', {...})`
    hardcoded locale for the finance dashboard's month/year subtitle. Must
    become locale-aware (use next-intl date formatting or pass the resolved
    cookie locale). If rendered under an EN locale, this would still show
    Arabic month names — a correctness bug, not just a cosmetic gap.

H2. app/dashboard/finance/recurring/new/page.tsx L62 — `dir="rtl"` HARDCODED
    on the page's root wrapper `<div>`. Under an EN/LTR locale this would force
    the entire "create recurring invoice" form into RTL layout regardless of
    the user's chosen language — a functional RTL/LTR-layout bug, not cosmetic.

H3. Massive STATUS_MAP / label-map duplication with NO shared accessor:
    the PO status map is copy-pasted 3× (list, list's SelectItems, detail page)
    with the SAME textual mismatches vs statuses.json (`po.acknowledged`,
    `po.received`, `po.invoiced` all differ from the local copies). Any i18n
    migration MUST first decide the canonical label per status THEN swap all
    3 copies atomically — swapping one copy to `useStatusLabels('po')` while
    leaving the other 2 as stale local literals would create a THIRD, worse
    inconsistency (some pages showing "مؤكد", others showing "تم الاستلام" for
    the exact same `acknowledged` status).

H4. The monthly/quarterly/yearly period-type/billing-cycle triad appears
    hardcoded independently in AT LEAST 6 places across 4 files with NO
    single shared entity in statuses.json that exactly matches (billingCycle
    has an extra `weekly`; leadBillingCycle has `one_time`+`annual` instead).
    A plan must decide: (a) reuse `billingCycle` and accept weekly is simply
    unused in these dropdowns, or (b) create a new entity. Blind accessor
    swap without this decision will either drop a needed option or import an
    unused one.

H5. Currency display-name maps (AED/USD/EUR/SAR → درهم/دولار/يورو/ريال) are
    hardcoded in 2 places (BasicInfo.tsx bare names, RevenueTargetForm.tsx
    names+codes) with a MINOR FORMAT INCONSISTENCY between them (one has the
    parenthetical currency code, one doesn't) — no existing statuses.json
    entity covers this; needs a new `finance.currencies.*` catalog to avoid
    a third divergent copy being created during migration.

H6. Recharts formatter-callback Arabic strings (section 16) are invisible to
    naive JSX-scanning migration scripts — explicit manual pass required for
    the 3 chart files, or the migration will silently miss 6+ hardcoded
    strings (المبلغ ×2, الإيرادات ×2, المصاريف ×2) plus 4 hardcoded 'en-AE' +
    literal 'AED' occurrences.

H7. Two separate "delete confirmation" AlertDialog description strings
    (purchase-orders/[id]/page.tsx L243, suppliers/[id]/page.tsx L107) are
    NEAR-identical templated sentences ("هل أنت متأكد من حذف {noun} هذا؟ لا
    يمكن التراجع عن هذا الإجراء.") that should collapse into ONE parametrized
    ICU message (`{entityLabel}` placeholder) rather than 2 (soon-to-be-N as
    more entities get delete confirms) hand-copied variants — same class of
    duplication CLAUDE.md's DRY conventions elsewhere warn against.

H8. cards/page.tsx is a documented ORPHANED page (Finance Remediation decision:
    "Subscriptions module SUNSET... Cards page is now orphaned — removal
    candidate"). Spending i18n-migration effort on 290 lines of a page flagged
    for removal is likely wasted work — the Phase-4 plan should explicitly
    decide skip-and-flag-for-deletion vs full-migrate before scheduling this
    file's work.

H9. Raw `fetch()` + manual `useState`/`useEffect` pagination (NOT React Query)
    is used throughout purchase-orders-client.tsx, suppliers-client.tsx,
    purchase-orders/[id]/page.tsx, suppliers/[id]/page.tsx, cards/page.tsx
    (partial), and client-statement/[clientId]/page.tsx — a pre-existing
    CLAUDE.md violation (mandatory React Query data layer) that is orthogonal
    to i18n but will make wiring `useTranslations()` + reactive locale-switch
    re-fetching MORE AWKWARD in these files than in the already-compliant
    recurring/page.tsx (which uses `useRecurringInvoices` + `useMutation`).
    Not this scan's job to fix, but the i18n plan should note it raises
    implementation friction in 6 of 14 in-scope files.

H10. RecurringSummary.tsx L44 references button text "توليد الفواتير المستحقة"
     that does NOT match the actual button label in recurring/page.tsx L59
     ("توليد" / "جاري..." while pending) — a PRE-EXISTING content drift bug
     (not caused by i18n) that will get "frozen into two different catalog
     keys" if migrated as-is without a content review flag. Recommend the
     plan include a one-line content-parity fix alongside the string extraction.
