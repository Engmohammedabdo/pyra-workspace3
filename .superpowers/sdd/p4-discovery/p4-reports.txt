HAZARDS (exported artifact / server-computed label byte-changes for AR users)
================================================================
H1. **CSV export headers + Yes/No cell values are 100% hardcoded Arabic** (app/api/reports/export/route.ts, §30). Every exported CSV (projects, clients, revenue, team, storage, expenses, invoices, finance) has Arabic column headers baked directly into `toCSV()` calls, and clients export additionally bakes Arabic literal VALUES ('نعم'/'لا') into data cells. If Phase-4 makes the dashboard UI bilingual but leaves this route untouched, an English-locale admin downloading "تصدير CSV" gets an all-Arabic file — a stronger form of the CLAUDE.md rule "Recipient-language rendering... pass an explicit locale" (this note in CLAUDE.md is about notifications/emails, but the SAME principle applies to exported files a user actively requests in their own UI language). DECISION NEEDED: (a) thread the current UI locale into `/api/reports/export?...&locale=en`, producing fully-English CSVs for EN users, OR (b) keep CSV exports Arabic-only forever (defensible since these are often shared with UAE accountants/regulators who expect Arabic) and mark this route `i18n-exempt` explicitly in code comments. Recommend (a) for headers (cheap — swap a header array) but flag that (a) also requires resolving H2 below (the 'نعم'/'لا' hardcoded cell value) and H3 (raw untranslated enum values already leaking into the CSV, e.g. `p.status` in invoices export is a raw DB enum like "sent"/"overdue" that's ALREADY effectively English abbreviation-ish — inconsistent bilinguality even before Phase-4 touches anything).

H2. **`.toLocaleDateString('en-CA')` × 6 in the export route is a GOOD hardcode that must survive the i18n pass unchanged** — these calls exist for CSV machine-readability (ISO-sortable dates), NOT for display polish. A naive "make all toLocale* calls locale-aware" sweep would WRONGLY localize these into Arabic-formatted dates inside a CSV, breaking spreadsheet re-import / sorting for the finance team. Explicit exemption needed, ideally as an inline comment at each of the 6 call sites (or refactor into one shared `csvDate()` helper with a comment once, replacing all 6).

H3. **Server-computed month/quarter labels baked into finance report API payloads** (pnl/cashflow/vat routes, §12-14) mean the chart x-axis labels, table row labels, and CSV-adjacent report body text are ALL pre-rendered in Arabic before reaching the client. Unlike the CRM dashboard's `label_ar`/`message_ar` fields (Phase-3, which had a natural client-side alternative via `useStatusLabels`), there is NO existing client-side month-name catalog being bypassed here — `MONTH_NAMES_AR` genuinely has no English counterpart yet in `lib/constants/dates.ts`. Fixing this requires EITHER adding `MONTH_NAMES_EN` + threading a locale param through 3 API routes (pnl/cashflow/vat) + the ad-hoc `arabicQ` quarter array in pnl/route.ts (§12), OR moving month/quarter-label formatting entirely client-side using `start`/`end` ISO dates + `Intl.DateTimeFormat(locale, {month:'long', year:'numeric'})`. The aging/route.ts `label`/`labelEn` pattern (§17) is the ONLY existing precedent for bilingual server labels in this whole surface and should inform whichever approach is chosen.

EXPLICIT ANSWER — does app/dashboard/reports belong to finance or is it a separate module?
RECOMMEND: DEFER (treat as a separate module, migrate in its own pass, NOT bundled into `finance.reports.*`).
Rationale:
  1. Permission gate differs: app/dashboard/finance/reports/page.tsx sits under the Finance module (implicitly gated by finance.view via its API routes), while app/dashboard/reports/page.tsx explicitly gates on requirePermission('reports.view') — a DIFFERENT permission (`reports.view`, not `finance.view`) covering projects/clients/storage/revenue/team, i.e. an org-wide operational-reporting module, not a finance-specific one.
  2. Sidebar/IA separation: per CLAUDE.md's architecture map, `/dashboard/finance/reports` is a Finance sub-page while `/dashboard/reports` is its own top-level "التقارير" section (confirmed by reports-client.tsx's own <h1>التقارير</h1> + distinct card grid linking to /dashboard/reports/{projects,clients,storage,revenue,team}) — these are TWO DIFFERENT IA nodes in the sidebar, not nested.
  3. Overlapping vocabulary (revenue/clients/team language) is coincidental domain overlap, not architectural coupling — the catalog can still SHARE common keys (`common.noDataToDisplay`, `common.reportLoadFailed`, `common.backToReports`, `common.overdue`) without merging the two modules' dedicated namespaces.
  4. Practical migration sequencing: bundling both in one pass roughly DOUBLES this phase's file count (24 files total, 8 of which are the general-reports surface: page.tsx/reports-client.tsx + 5 sub-pages) — cleaner to close out finance.reports.* first (13 files: finance page + 5 finance-report-tab components + shared SummaryCard/DateRangeFilter + 5 finance API routes, +2 shared components/reports/* used by BOTH) then do a follow-up pass for reports.* (8 files: general reports page/client + 5 sub-pages + 6 API routes, the export route is genuinely SHARED infrastructure used by both).
  If the planner prefers a SINGLE combined phase for efficiency (since components/reports/ExportButton.tsx, ReportChart.tsx, ReportDateFilter.tsx are ALREADY shared infrastructure consumed by both surfaces), that is a defensible INCLUDE decision too — flagging both options with a recommendation for DEFER/separate-pass as the primary recommendation given the permission-gate and IA evidence above.
