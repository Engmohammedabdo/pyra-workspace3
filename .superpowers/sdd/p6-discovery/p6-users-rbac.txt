### rbac.ts STRUCTURE (for the plan to catalogue) ###
`PERMISSION_MODULES: PermissionModule[]` — a flat ARRAY of 33 module
objects. Each module: `{ key: string, label: string, labelAr: string,
permissions: { key: string, label: string, labelAr: string }[] }`.
Nested one level: module → permissions array. 33 modules × ~1-12 perms each
= ~156 English/Arabic label pairs (123 permission-level + 33 module-level).
This is a DISPLAY-NAME catalogue only — it does NOT gate anything at
runtime (the actual `PERMISSIONS` const + `hasPermission()`/
`buildUserPermissions()` engine above it uses only the `.key` strings; the
`label`/`labelAr` fields exist purely so 3 UI surfaces can render
human-readable checkboxes/badges).

USAGE — confirmed UI-ONLY (grep across whole repo), never server-side:
  - app/dashboard/roles/roles-client.tsx (role-editor permission checklist,
    module.labelAr for group headers + perm.labelAr for individual boxes)
  - app/dashboard/users/users-client.tsx (per-user "صلاحيات إضافية" /
    extra_permissions checklist, same module.labelAr + perm.labelAr pattern)
  - app/dashboard/profile/profile-client.tsx (OUT OF SCOPE for this census,
    but also reads PERMISSION_MODULES for a read-only "my permissions"
    display — same UI-only usage)
  No `app/api/**` route reads `.label`/`.labelAr` — only `.key` (confirmed
  by the `PERMISSION_MODULES` grep hit list: only the 3 files above + docs).

PROPOSED NAMESPACE (for the plan to evaluate, not decided here):
  A `rbac.json` namespace keyed by module key → permission key, e.g.
  `rbac.modules.dashboard.label` / `rbac.modules.dashboard.permissions.view`,
  NOT the current array-of-objects shape. See CRITICAL HAZARD below —
  `PERMISSION_MODULES` is a literal TS array; if migrated naively into a
  next-intl JSON catalog as an array it violates the CLAUDE.md rule
  "Catalogs must NEVER contain JSON arrays." The catalog and the TS runtime
  array must be decoupled: TS keeps `key` order/structure (for iteration),
  JSON catalog holds only `<module.key>.label` / `<module.key>.perms.<perm
  suffix>.label` as keyed strings, and the component zips them back together
  by key at render time (same pattern as `lib/i18n/status-labels.ts`).
