ARRAY HAZARD detail: per CLAUDE.md i18n rules ("Catalogs must NEVER contain JSON
arrays — arrays poison next-intl's type inference. Use keyed objects instead") AND
per `lib/i18n/messages.ts` deepMerge doc comment (line 23: "Arrays are replaced
wholesale, not merged") — if tips/keywords were naively dumped into a next-intl
JSON catalog as `string[]`, they would (a) poison next-intl's typed-key inference
(the same problem the `calendar.dayNames` object-not-array fix already solved
elsewhere per CLAUDE.md), AND (b) break the "EN merged over AR, missing EN key
renders Arabic" per-leaf fallback guarantee — a partially-translated EN tips array
would wholesale-replace the whole AR tips array in deepMerge, not fall back item-
by-item. Any plan MUST key both arrays as index-keyed objects, e.g.:

  tips: { "0": "...", "1": "...", "2": "..." }
  keywords: { "0": "...", "1": "..." }

so each index is an independent leaf that deepMerge can fall back on individually.
Consumers (page-guide.tsx, module-card.tsx, searchModuleGuides) currently do
`.map()`/`.some()`/`.length` over real arrays — the migration must either (a) keep
a runtime `string[]` shape by doing `Object.values(keyedObject)` after i18n lookup,
or (b) refactor consumers to iterate `Object.entries()`. Recommend (a) — preserves
every existing `.map((tip, i) => ...)` call site untouched.

7. keywords[] CONSUMPTION — SEARCH BEHAVIOR (searchModuleGuides, lines 1171-1183)
---------------------------------------------------------------------
  export function searchModuleGuides(query?: string): ModuleGuide[] {
    const all = Object.values(MODULE_GUIDES);
    if (!query || query.trim() === '') return all;
    const q = query.toLowerCase();
    return all.filter(
      (m) =>
        m.description.includes(q) ||
        m.descriptionEn.toLowerCase().includes(q) ||
        m.goal.includes(q) ||
        m.keywords.some((k) => k.includes(q))
    );
  }

Consumed by app/dashboard/guide/page.tsx:174 (the search box) — this is the ONLY
call site in the codebase (grep-confirmed no other importer of searchModuleGuides).

Hazards in the search logic itself (relevant to the migration plan, not just the
data):
- `q = query.toLowerCase()` is applied to the query, but `m.description`,
  `m.goal`, and each keyword `k` are compared via `.includes(q)` WITHOUT
  `.toLowerCase()` on the Arabic-bearing side (only `descriptionEn` gets
  `.toLowerCase()`). Arabic has no case concept so this is harmless for Arabic
  text, but for the ENGLISH keywords already mixed into `keywords[]` (e.g.
  'dashboard', 'kanban', 'pipeline') an uppercase query like "Dashboard" WILL
  match today only because `q` itself is lowercased and English keyword strings
  in the source are already lowercase by convention — a case-SENSITIVE keyword
  entered in Title Case in the data would silently stop matching. Not a bug to
  fix in this census, but worth the plan noting since post-migration EN keywords
  will multiply.
- Spec §14 risk (per task prompt): "the plan must keep Arabic keywords AND add
  English keywords." Current `keywords[]` arrays ALREADY do this informally and
  inconsistently — some entries mix EN+AR in one array (e.g. '/dashboard' line 39:
  `['dashboard', 'الرئيسية', 'إحصائيات', 'ملخص']`), others are EN-heavy (e.g.
  '/dashboard/roles' line 793: `['roles', 'أدوار', 'صلاحيات', 'RBAC', 'permissions',
  'wildcard', 'BASE_EMPLOYEE']` — 4 of 7 items are English/technical terms already).
  This means the migration is NOT starting from a clean Arabic-only keyword corpus
  — it must first classify each of the 499 existing keyword items as
  Arabic-searchable vs English-searchable vs technical-literal (e.g. 'RBAC',
  'BASE_EMPLOYEE', 'API' are code/config terms, not language-translatable at all)
  before deciding what an "EN keyword set" even adds on top. Recommend the plan
  treat `keywords` as a SINGLE flat locale-agnostic search corpus (not something
  that needs an AR variant and an EN variant at all) rather than forcing it
  through the description/goal/tips translation model — the search function
  already searches across ALL keywords regardless of locale; keying it per-locale
  would require running search over BOTH locale keyword sets to preserve current
  find-by-Arabic-or-English behavior, which is extra complexity for a field that's
  already functioning as a mixed-language index rather than display copy.


8. BILINGUAL / LOCALE HAZARDS — SUMMARY
---------------------------------------------------------------------
(a) ARRAY HAZARD (tips[], keywords[]) — see §3. Must become index-keyed objects
    in any next-intl catalog; runtime consumers need `Object.values()` shimming
    or a refactor to `Object.entries()`.
(b) descriptionEn is currently used as an UNCONDITIONAL English display value in
    2 render sites (page-guide.tsx:66, module-card.tsx:30) regardless of active
    locale — a genuine pre-existing locale bug/design-decision that predates this
    census and must be explicitly resolved (not silently preserved) by the plan.
(c) `guide/page.tsx` has ZERO existing next-intl wiring — no `useTranslations`/
    `getTranslations` import at all. It is a full-file migration, not an
    incremental one.
(d) `page-guide.tsx` is a HYBRID — its own chrome is migrated (Phase-something
    already shipped `common.guide.*`), but the DATA it renders (goal/tips from
    MODULE_GUIDES) is not. A partial re-migration, not a green-field one.
(e) `module-card.tsx` duplicates a string ("نصائح الاستخدام") that already exists
    as `common.guide.tips` — the plan should de-duplicate onto the existing key
    rather than minting a second key with identical English/Arabic values.
(f) Referential coupling: `SECTIONS[].hrefs` (guide/page.tsx) must stay
    byte-identical to `MODULE_GUIDES` object keys/hrefs, or entries silently
    vanish from the grouped view (no error, no warning — `.filter(Boolean)`
    swallows the mismatch). Any key-renaming during migration (e.g. switching
    object keys to sanitized identifiers for the i18n catalog) must be mirrored
    in SECTIONS.hrefs in the same commit.
(g) `getModuleGuide()` prefix-matching (module-guide.ts:1153-1166) walks the
    PATHNAME (a URL, not the catalog key) — this logic is entirely
    locale-independent and structural; no hazard, just noting it must keep using
    `href`/object-key route strings, never a translated display string.
(h) File-size: module-guide.ts is 1184 lines — by far the largest single file
    touched in the whole i18n effort to date (guide/page.tsx 262, page-guide.tsx
    122, module-card.tsx 65 — all under the 300-line CLAUDE.md convention target
    except module-guide.ts itself, which is a pure data file, not a component, so
    the 300-line component-split rule doesn't strictly apply — but the SIZE is
    still the reason this census/plan phase exists rather than a quick inline fix).
(i) Precedent pattern exists and should be followed: `components/layout/
    nav-config.ts` (233 lines) already solved an identical "static TS config with
    embedded Arabic display strings" problem for the sidebar. Its pattern: the
    config file keeps ONLY structural fields (href, icon, permission, badgeKey)
    plus a typed `key` field; `NavItemKey = keyof typeof nav.nav.items` derives a
    literal-string union straight from the AR catalog import
    (`import type nav from '@/messages/ar/nav.json'`), and render sites call
    `t(\`items.${item.key}\`)`. `messages/ar/nav.json`'s `nav.groups.*` shape is
    confirmed to be pure keyed objects (`{ title, description }`), NO arrays
    anywhere in that file — module-guide would be the FIRST namespace in the
    project needing the keyed-object-for-arrays workaround, since nav.json never
    needed one.
(j) `lib/i18n/messages.ts` NAMESPACE_FILES (line 11-15) is currently:
    ['common','nav','auth','statuses','mywork','boards','calendar','api','crm',
    'finance','hr'] — tracked through "Phase 5 adds 'hr'". A module-guide
    namespace would be net-new (e.g. 'guide' — but note `common.guide.*` already
    exists for the popover chrome, so the plan must decide: extend `common.json`'s
    existing `guide` key, or add a distinct new namespace file+entry to avoid
    `common.json` ballooning by ~1100 string values for what is fundamentally a
    different (per-module reference content) concern than shared UI chrome).
