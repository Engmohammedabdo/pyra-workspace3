Hazard-ADJACENT findings (not i18n, but flagged for a separate follow-up — NOT blocking this census; consolidated from all 4 sub-scope agents):
- **Sun–Thu weekend fallback regression (2 independent sites)** — CLAUDE.md's locked Batch C decision states Pyramedia's weekend is Sunday-only (`DEFAULT_WORK_DAYS = [1,2,3,4,5,6]` in `lib/constants/auth.ts`), yet TWO fallback code paths still hardcode the OLD Sun–Thu week: `app/api/dashboard/attendance/summary/route.ts:65` (`let workDays: number[] = [0, 1, 2, 3, 4]; // Default: Sun-Thu`) and `app/api/timesheet/route.ts:58-66` (`UAE_DEFAULT_SCHEDULE.work_days = [0, 1, 2, 3, 4]`). Both are reached only when a user has no `work_schedule_id` and no default schedule resolves — an edge case, but a real business-logic drift contradicting the explicit CLAUDE.md warning "the Sun–Thu fallback must not come back."
- **Duplicated manual UTC+4 offset math instead of `dubaiDayKey()`** (3 sites, all in attendance): `app/api/dashboard/attendance/route.ts:83-87`, `clock-out/route.ts:25-28`, `summary/route.ts:94-96` — each independently computes `new Date(nowUtc.getTime() + uaeOffset)` then `.toISOString().slice(0,10)` instead of calling the canonical `dubaiDayKey()` helper (Phase 15.1 lock). Currently correct (Dubai has no DST) but a drift risk and a DRY violation — 3 copies of the same arithmetic.
- **UTC-day-key smell in timesheet week-range filter** — `app/api/timesheet/route.ts:48` uses `startDate.toISOString().split('T')[0]` / `endDate.toISOString().split('T')[0]` for a week-range filter. Lower severity than a same-day "today" cutoff but still inconsistent with the `dubaiDayKey()` standard. By contrast `app/api/leave/[id]/route.ts:218-219`, `app/api/my-productivity/route.ts:20`, `app/api/hr/overview/route.ts` (multiple sites), `app/api/hr/leave-balances/route.ts:76`, and `app/api/hr/productivity/route.ts:20` all correctly use `dubaiDayKey()` — good reference patterns.
- **Legacy v1 `pyra_leave_balances` table still read** — `app/api/leave/balance/route.ts:76-93` (approximately) contains a fallback code path that queries the DEAD legacy `pyra_leave_balances` table, directly contradicting the Batch D lock ("`pyra_leave_balances_v2` is the SINGLE source of truth... do NOT read/write" the v1 table). Both tables were reportedly 0-row at cutover so this is inert in practice, but it's live dead-code that should be removed as cleanup — flagged per the task's explicit instruction to note any v1-table reference.
- **Overtime summary missing `status='approved'` filter** — `app/api/dashboard/overtime/summary/route.ts:39-44` filters `.eq('is_overtime', true)` but does NOT additionally filter `.eq('status', 'approved')`, diverging from the documented payroll overtime formula (CLAUDE.md Payroll Integrity Fixes #3: "approved timesheets (is_overtime=true AND status='approved')"). This may include draft/pending overtime hours in the summary's `estimated_pay` figure. Zero Arabic-string content — pure business-logic observation, not a class-(d) hazard.

- No hardcoded `ar-EG` locale calls (`.toLocaleDateString('ar-EG', ...)`) found anywhere in this census's 44 files (contrast with the Finance census's `finance/dashboard/route.ts:147` finding).

================ LEAVE-TYPE-NAME-MATCHING HAZARD (mandatory dedicated section — investigated explicitly per task instructions) ================

Per CLAUDE.md's locked HR/Payroll decisions: `pyra_leave_requests.type` is a NAME string matching `pyra_leave_types.name` — there is NO `leave_type_id` column on `pyra_leave_requests`. Every site below resolves `is_paid` / balance behavior by comparing this NAME string, not a stable ID. **These are DB match keys, not UI strings, and must NEVER be run through the i18n translation layer or any automated find-and-replace during the Phase 5+ catalog migration** — `pyra_leave_types.name` values are live comparison keys whose exact byte value drives balance-deduction and is_paid-detection business logic. Confirmed by direct orchestrator re-grep (`\.eq\('name'` across app/api/leave/**) — exactly 3 matches, matching the discovery agent's report precisely.

**Site 1 — `app/api/leave/route.ts:85`** (POST — new leave request):
```
.from('pyra_leave_types')
.select('id, default_days, is_paid')
.eq('name', type)              // line 85 — `type` is the raw request-body field (line 54)
.eq('is_active', true)
.single();
```
`type` is caller-supplied, expected to exact-match a `pyra_leave_types.name` row (which may itself be an Arabic string, e.g. a custom Arabic-named leave type). **(d)-ADJACENT COMPARISON HAZARD — DB match key, must NEVER be translated.** Downstream: `leaveType?.is_paid` gates the balance-sufficiency check.

**Site 2 — `app/api/leave/[id]/route.ts:65`** (PATCH — approve branch, balance deduction):

**(d)-ADJACENT COMPARISON HAZARD.** Downstream: `leaveType.is_paid` gates whether `used_days` increments in `pyra_leave_balances_v2` on approval.

**Site 3 — `app/api/leave/[id]/route.ts:228`** (PATCH — reason-based cancel branch, balance restoration):
