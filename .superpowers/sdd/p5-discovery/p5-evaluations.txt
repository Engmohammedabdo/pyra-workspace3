9. HAZARDS section (business logic branching / status-vs-comparison risk)

H1. [MEDIUM — statuses entity mismatch] The existing `evaluation` entity in
messages/ar/statuses.json (`{draft: 'مسودة', in_progress: 'جاري التقييم', completed: 'مكتمل'}`)
does NOT match either of the TWO real status domains this module actually uses:
  - EvaluationPeriod.status: draft / active / closed
  - Evaluation.status: draft / submitted / acknowledged
Only the `draft` key overlaps (and even then, the catalog's `draft` label 'مسودة' happens to match
this module's STATUS_LABELS.draft — coincidental agreement, not a wired connection). The catalog's
`in_progress` and `completed` values are UNUSED by this module; this module's `active`/`closed`/
`submitted`/`acknowledged` are UNDEFINED in the catalog. This is a NAMING COLLISION requiring an
explicit plan decision: (a) rename/extend the existing `evaluation` entity to cover both real status
sets (likely needs splitting into `evaluationPeriod` + `evaluation` as TWO separate entities since
they're different DB columns on different tables), or (b) leave `evaluation` in statuses.json
alone (maybe it's used by content-pipeline-adjacent code elsewhere?) and introduce fresh entities
for this module. Recommend verifying grep-wide whether `evaluation` entity from statuses.json is
consumed ANYWHERE currently (this scan found ZERO consumption of `useStatusLabels('evaluation')` or
`EVALUATION_STATUS_LABELS` import in the 6 files scanned) before deciding whether to repurpose or
introduce new entities. This is a pre-existing catalog/implementation drift, not something this
i18n pass introduced.

H2. [MEDIUM — bonus-recommendation notification content, Phase-8-exemption question] The task
asked specifically whether the bonus-recommendation strings (tiered 5/10/15%) are Phase 8 exempt
(the "notification content stays Arabic" carve-out per CLAUDE.md's i18n rules: "Notification /
WhatsApp / DB-data strings inside routes STAY Arabic ... until Phase 8"). Findings:
  - The CLIENT-SIDE UI in evaluations-client.tsx does NOT contain the tiered percentage text
    itself — the 5/10/15% tiering logic + message construction lives SERVER-SIDE (per CLAUDE.md's
    "HR Gap-Remediation" section: "the `recommend_bonus` action ... is now surfaced as a button").
    The client only renders `data?.message || 'تمت التوصية بمكافأة بنجاح'` (L371) — i.e. it PREFERS
    the server's message and falls back to a generic client string.
  - This IS the exact "server response message" pattern CLAUDE.md's i18n rules cover via the `api`
    namespace (`getTranslations('api')` in route handlers) — the underlying
    `/api/dashboard/evaluations/[id]` PATCH route (action=recommend_bonus) is OUT OF SCOPE for this
    census (route handlers are a separate discovery pass per the api-namespace convention), but the
    CLIENT'S fallback string ('تمت التوصية بمكافأة بنجاح' at L371, and 'فشل في التوصية بمكافأة' at
    L375) ARE in scope as ordinary UI toast strings — they should go through the `hr.evaluations.*`
    (or api) catalog like any other toast, they are NOT the tiered-percentage content itself.
  - CONCLUSION: the tiered 5/10/15% bonus AMOUNT text (if it appears in a notification body sent to
    the employee) is NOT present anywhere in these 6 files — it must live in the API route or in
    `lib/notifications/`. This client-side census found no percentage literals to exempt or migrate.
    Recommend the API-route discovery pass for `/api/dashboard/evaluations/[id]` explicitly checks
    for the tiered percentage strings and applies the Phase-8 notification-content exemption there,
    not here.
  - The numeric GATE (`overall_rating >= 3.5`, L732) that shows/hides the bonus button is untouched
    by any of this — pure numeric comparison, no i18n risk.
