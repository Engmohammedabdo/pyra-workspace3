5. HAZARDS section
================================================================

H1. [HIGH — locale-bypass on live/high-visibility displays] TodayClockCard.tsx L50-63: the live current-time + current-date display (ticks every second via the parent's `setInterval`) uses raw `toLocaleTimeString('ar-AE', ...)` / `toLocaleDateString('ar-AE', ...)` instead of the shared `formatTime()`/`formatDate()` helpers that already accept a `locale` param. This is the MOST VISIBLE clock on the page and currently has NO path to render in English without a code change beyond simple string-catalog work — it needs its own refactor to either call the shared helpers (if they can be extended to support this exact option set: weekday/day/month long-form + separate time-with-seconds) or take an explicit `locale` param mirroring the Phase 0-2 convention.

H2. [MEDIUM — component-level dir override, not a string swap] AdminAttendanceDialog.tsx L115 `dir="rtl"` on `<DialogContent>` and timesheet-client.tsx L462 `dir="rtl"` on `<Tabs>` are HARDCODED full-component directional overrides (distinct from the "keep-as-is" numeric-field dir="ltr" pattern documented in Phase-4). Once the global `<html dir>` becomes locale-conditional per CLAUDE.md's i18n architecture, these two hardcoded `dir="rtl"` props will fight the page's real direction in EN-locale sessions, breaking RTL-specific components (Dialog close button position, Tabs underline direction) for English users. Needs an explicit decision: make these conditional on the resolved locale, or remove the override and let it inherit from `<html dir>` (verify Dialog/Tabs don't need an explicit dir at all — check if the override was ever necessary vs copy-paste).

H3. [MEDIUM — reused Arabic day-name array, reuse existing catalog instead of duplicating] AttendanceCalendar.tsx's local `DAY_NAMES_AR` (L17) duplicates content ALREADY present in `messages/ar/calendar.json`'s `calendar.dayNames` keyed object. Per the locked i18n rule "Catalogs must NEVER contain JSON arrays — use keyed objects instead (see calendar.dayNames...)" — this is exactly the precedent to follow. Recommend either importing/consuming the existing `calendar` namespace here, or replicating the exact same keyed-object shape under a new key if cross-namespace `useTranslations` composition isn't the established convention (verify against how other modules that need calendar-adjacent data have handled this — check if any other non-calendar page already imports the `calendar` namespace for day names before deciding the pattern).

H4. [LOW-MEDIUM — undocumented 3rd 'ar-EG' locale exception] timesheet-client.tsx L493 uses `toLocaleDateString('ar-EG', ...)` for the entry's short weekday name, while every OTHER Arabic locale call in this exact census scope uses 'ar-AE' (TodayClockCard x2, AttendanceCalendar x1). The CLAUDE.md HR/Payroll v1.1 cleanup section documents exactly TWO intentional 'ar-EG' exceptions elsewhere (`calendar-event-pill.tsx`, `crm/dashboard/dashboard-data-sources.tsx`) with an explicit "MUST NOT be swapped" lock — this timesheet call site is NOT among those two, so it's either (a) an unnoticed 3rd intentional exception that should be documented alongside the other two, or (b) an inconsistency that should conform to 'ar-AE' like its sibling files. Flag for the plan owner to decide before touching this line — do NOT silently normalize it without confirming which case applies.

H5. [LOW — stale weekend-fallback pre-existing bug, unrelated to i18n but adjacent] attendance-client.tsx L119: `workSchedule?.work_days ?? [0, 1, 2, 3, 4]` (Sun-Thu, 0-indexed) contradicts the locked "Pyramedia weekend = Sunday only" / `DEFAULT_WORK_DAYS = [1,2,3,4,5,6]` decision from the Batch-C work-schedules lock. This is a DATA/LOGIC bug the i18n pass will touch (same file, same function) but must NOT silently "fix" as a side effect of string-catalog work — flag to the user/plan-owner as a separate concern, consistent with how Phase-4 flagged the contract `in_progress` / expense `draft` status gaps as "pre-existing, not ours to silently fix."

H6. [LOW — two new statuses-catalog entities needed] Timesheet has TWO separate status vocabularies used in this scope with NO shared-constant backing at all (unlike attendance, which is fully ready via `ATTENDANCE_STATUS_LABELS` + the existing `statuses.attendance` JSON entity): (a) entry-level `STATUS_LABELS` = draft/submitted/approved/rejected, matching the existing bare `TIMESHEET_STATUS` enum in lib/constants/statuses.ts but with NO `TIMESHEET_STATUS_LABELS` map ever having been created; (b) period-level `PERIOD_STATUS_LABELS` = open/submitted/approved/rejected (note: "open" replaces "draft" as the initial state — a genuinely different enum, not just a duplicate). The i18n plan needs to author BOTH `timesheet` and `timesheetPeriod` entities in statuses.json from scratch (no existing Arabic canonical source to extract from except these two local consts in timesheet-client.tsx itself — verified as the ONLY place these labels are defined, so extraction is straightforward, just needs the plan to explicitly create 2 new entities rather than assuming reuse of an existing one).
