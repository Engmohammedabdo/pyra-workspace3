6. HAZARDS section
================================================================

H1. [HIGH — money discipline / DRY violation] app/dashboard/my-payslips/my-payslips-client.tsx
L69-71 defines a LOCAL `formatCurrency()` function (LRM-mark + `toLocaleString('en-US', ...)` +
manual currency-code suffix) instead of importing the canonical `formatCurrency` from
`@/lib/utils/format` used everywhere else in the payroll scope (PayrollRunRow.tsx,
EmployeePaymentsTab.tsx both import the shared one). All 12 call sites in this file use the local
shadow. Every call site DOES pass an explicit currency (compliant with the multi-currency-payroll
money-discipline lock in spirit), but the DUPLICATE IMPLEMENTATION itself is a hazard — if the
canonical formatter's behavior (rounding, symbol placement, RTL handling) changes in the future,
this file silently diverges. Recommend the i18n pass ALSO swaps this to the shared import as a
drive-by fix, since the file is already being touched for string-catalog work and the two
implementations currently produce visually similar but not-guaranteed-identical output.

H2. [MEDIUM — data-model inconsistency, not purely i18n] Payment `source_type` label maps exist
as 3 SEPARATE inline Record literals (EmployeePaymentsTab.tsx 7 keys, my-payslips-client.tsx 6
keys, AddPaymentDialog.tsx's SelectItem list 5 keys) with 3 DIFFERENT subsets of the same
underlying set {task, bonus, deduction, salary, advance, commission, overtime}. The label TEXT is
identical across all 3 where keys overlap (no wording drift), but AddPaymentDialog's create-form
does not expose `salary` or `advance` as selectable options — meaning those payment types can only
ever be created by some other code path (not found in this scope — likely a server-side-only
source, e.g. the payroll calculate route synthesizing `source_type='salary'` items, or a legacy
data path). Flag to the plan/user: decide whether to (a) consolidate into ONE shared constant +
new `paymentSourceType` statuses.json entity with all 7 keys, keeping the create-form's narrower
selectable subset as an intentional UI restriction, or (b) treat the 5-vs-7 gap as a product bug
to fix. Recommend (a) — the display maps should share one source of truth; the CREATE form's
option list can justifiably stay a curated subset (users shouldn't manually create "salary" type
payments, those come from payroll runs).

H3. [MEDIUM — cross-entity status-label collision] app/dashboard/my-payslips/
my-payslips-client.tsx's single `STATUS_LABELS` object (L61-67) is used at TWO call sites for TWO
DIFFERENT status entities: L285 renders `payslip.run_status` (a `payroll` entity value: draft/
calculated/approved/paid) and L378 renders `p.status` (an `employeePayment` entity value:
pending/approved/paid/rejected). The map's `pending` key only makes sense for the employeePayment
use-site (payroll runs never have a `pending` status in the catalog) — this "works" today only
because Arabic object-key lookup silently falls through to `|| payslip.run_status` / `|| p.status`
raw-value fallback when a key is absent for the OTHER entity's vocabulary. When migrating to
`useStatusLabels(entity)`, this file needs TWO separate hook calls
(`useStatusLabels('payroll')` for L285, `useStatusLabels('employeePayment')` for L378) — a
straight 1:1 replace of the single local map would be WRONG and must not be attempted.

H4. [LOW — server-message passthrough coupling] PayrollRunRow.tsx L81 `toast.warning(w)` for each
string in `result.warnings` (from the calculate-payroll mutation response) renders live
server-supplied Arabic text with no client-side fallback/wrapper. This is a stronger coupling than
Phase-4's `json.error || 'fallback'` pattern (there's no fallback here at all — it's 100% server
text). Until `/api/dashboard/payroll/[id]/calculate`'s warning strings are migrated to the `api`
i18n namespace, an EN-locale user calculating payroll will see Arabic warnings inline among
otherwise-English UI. Sequencing dependency for the phase-5 API-route census.

H5. [LOW — Phase 9 PDF exemption, noted for cross-reference] `lib/pdf/payslip-pdf.ts` (252 lines,
OUT OF DIRECT SCOPE per this census's file list but INVOKED from both PayrollRunRow.tsx
(handleDownloadPayslip) and my-payslips-client.tsx (handleDownload)) contains hardcoded Arabic
labels for the payslip PDF line items: `'الراتب الأساسي'`, `'مدفوعات المهام'`, `'العمل الإضافي'`,
`'المكافآت'`, `'العمولات'`, `` `خصم ${idx+1}` ``, `'الخصومات'`, `'صافي الراتب'` — per CLAUDE.md's
i18n rules, PDF generators are Phase-9-EXEMPT (marked (pdf) not (ui)) and are NOT part of this
phase's string-catalog work. Listed here ONLY for completeness/cross-reference since the two
in-scope files both call into it — no action needed in Phase 5.

H6. [LOW — window.confirm vs AlertDialog] PayrollRunRow.tsx L117 uses native
`window.confirm(...)` for the delete-payroll-run confirmation, unlike Phase-4 finance pages which
uniformly use the shadcn `AlertDialog` component for delete confirmations. The Arabic string
itself is a normal translation target regardless, but flag the mechanism inconsistency for
awareness (native confirm() cannot be styled/RTL-controlled the same way, and its browser-default
text ["OK"/"Cancel"] won't route through next-intl at all — only the message body will).
