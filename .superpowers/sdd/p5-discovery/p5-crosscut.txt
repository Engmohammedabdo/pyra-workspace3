=== 10. HAZARDS (full list; top-3 called out in final summary) ===

1. **13 hardcoded 'ar-AE'/'ar-EG' toLocale*/Intl.* call sites across the HR
   surface** (§3) — far higher density than Finance's 1-of-7. Concentrated
   in components/attendance/{AttendanceCalendar,TodayClockCard}.tsx,
   components/payroll/{EmployeePaymentsTab,PayrollRunRow}.tsx,
   app/dashboard/my-payslips/my-payslips-client.tsx,
   app/dashboard/hr/onboarding/[id]/onboarding-detail-client.tsx,
   components/hr/onboarding/{OnboardingChecklist,WizardStepCompensation,
   WizardStepReview}.tsx, app/dashboard/timesheet/timesheet-client.tsx. Each
   needs an explicit per-site decision: thread the resolved UI locale
   (dates) vs. convert to a fixed locale-invariant format (money previews,
   matching formatCurrency's doctrine). A live attendance clock
   (TodayClockCard) rendering '11:42 صباحاً' unconditionally regardless of
   an English-locale user is the most user-visible instance.

2. **Wording collision across 3 files for EMPLOYMENT_TYPES/WORK_LOCATIONS
   enum labels** (§2) — onsite/contract/freelance each have 2-3 genuinely
   different Arabic words in production today (components/hr/onboarding/
   WizardStepPosition.tsx vs app/dashboard/users/[username]/user-detail-
   client.tsx vs app/dashboard/directory/directory-client.tsx). Only the
   FIRST file is in Phase 5's declared scope — the other two are carry-over
   hazards for a different phase, but if Phase 5 mints a new statuses.json
   `employmentType`/`workLocation` entity using its own file's wording, that
   entity will NOT match what the other two (still-unmigrated) files show,
   meaning the same real-world attribute displays 2 different words
   depending on which page the admin is viewing — a pre-existing UX bug the
   i18n effort will make more visible (same class as Phase 4's contract
   in_progress collision), not introduce.

3. **`pyra_leave_types.name`-matching + name_ar DB-driven display is NOT
   swappable to next-intl for v2 dynamic leave types.** app/dashboard/leave/
   leave-client.tsx:308 falls back to `dynamicBalances.find(b => b.name ===
   c.leave_type)?.name_ar || TYPE_LABELS[c.leave_type] || c.leave_type` —
   the v2 Batch-D leave-balances system (locked decision, CLAUDE.md) makes
   leave types ADMIN-CONFIGURABLE DB rows with their own `name`/`name_ar`
   columns (e.g. the seeded `lt_unpaid` row). The existing `leaveType`
   catalog entity (5 fixed keys: annual/sick/personal/unpaid/emergency) can
   only ever cover LEGACY/fixed types — any admin-added custom leave type's
   Arabic name is genuine DB content with NO English counterpart and NO
   catalog key, structurally identical to the CRM board-template-seeded-
   column exemption. Phase 5 must NOT attempt to route dynamic leave-type
   names through next-intl; keep the `name_ar || TYPE_LABELS[...] ||
   raw-name` fallback chain intact, and only swap the STATIC `TYPE_LABELS`
   shadow map (annual/sick/personal, lines 22-23 of leave-client.tsx) plus
   the STATUS_LABELS shadow (line 21, which spreads the canonical
   LEAVE_STATUS_LABELS + adds a local 'cancelled' override — verify at plan
   time whether the local 'ملغية' override diverges from canonical). Same
   applies server-side: app/api/leave/route.ts:85, app/api/leave/[id]/
   route.ts:65,228 all do `.eq('name', type)`/`.eq('name', existing.type)`
   against the DB — these are QUERY filters (not display), completely out
   of i18n scope, but confirm implementers don't accidentally "fix" the
   column name assumption while touching these files for unrelated reasons
   (per the project's own Engineering Discipline memory note about never
   guessing column names).
