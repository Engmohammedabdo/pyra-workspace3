# CRM System Audit — 2026-07-02

## Executive Summary

Adversarial, code-grounded audit of the CRM module (pipeline, leads, follow-ups,
customers/dossier, dashboard aggregators, crons, and the legacy `/dashboard/sales`
surfaces). **48 findings survived verification: 11 medium, 37 low.** No high/critical
issues remain after severity correction — the most damaging bugs are gated behind
admin-only or transient-failure preconditions, but several silently corrupt data,
block a core journey, or bypass an intended permission gate.

Recurring themes: (1) the **Supabase lazy-thenable `void <builder>`** anti-pattern is
still live in 6+ legacy sales-leads writes (audit/activity/notification/score/next_follow_up
never persist); (2) **currency-blind summation** mislabels EGP/USD/SAR as AED across
dashboard KPIs, dossier, and lead-detail cards; (3) **UTC-vs-Dubai day** math regressed
in 5 places despite the `dubaiDayKey()` lock; (4) **swallowed SELECT errors** in the
idle-check cron can emit false alerts.

**Top 5 most critical:**
1. **convert-to-customer returns 500 for any company-less (B2C) lead** — the won→customer
   journey is permanently blocked for an entire class of leads (`convert-to-customer/route.ts:186`).
2. **Reopening a closed_won lead leaves `is_converted`/`converted_at`/`win_probability`
   stale** — the deal keeps showing as a 100%-won customer and freezes forecast recompute
   forever (`move-stage/route.ts:189`).
3. **convert-to-customer partial write is unrecoverable** — a failed lead-link stamps
   `created:true` but leaves the lead unlinked, and every retry 409s (`convert-to-customer/route.ts:220`).
4. **sales_agent can set arbitrary `assigned_to` on POST leads/follow-ups**, bypassing the
   manager-only `leads.assign` gate (`crm/leads/route.ts:194`).
5. **Customer dossier undercounts revenue** — standalone (non-contract) invoices are
   excluded from LTV/payment-health, and mixed-currency sums are mislabeled AED, so a paying
   customer can render as 0-revenue / at-risk (`dossier/route.ts:227`).

## Findings by Area

### Security & RBAC

#### [medium] POST /api/crm/leads & /api/crm/follow-ups accept arbitrary `assigned_to`, bypassing the manager-only `leads.assign` gate — `app/api/crm/leads/route.ts:194`
- **What's wrong:** On create the handler writes `body.assigned_to` verbatim, gated only by
  `leads.create` (which sales_agent holds). Reassignment is elsewhere privileged: PATCH
  `/api/crm/leads/[id]:180` rejects an `assigned_to` change without `leads.assign`, and
  `leads.assign` is intentionally omitted from sales_agent (`rbac.ts:836-838`). The same
  ungated pattern exists in POST `/api/crm/follow-ups:137-139`. There is no validation that
  the target username is a real, active, lead-capable user.
- **Failure scenario:** Agent `sayed` POSTs `{name, phone, assigned_to:'elharm'}`. The lead is
  created owned by another agent — inflating `elharm`'s `assigned_to`-grouped dashboards while
  the `leads.assign` control is never consulted. `assigned_to:'nonexistent'` orphans the lead
  (no non-admin scope can ever see it).
- **Suggested fix:** If `body.assigned_to` differs from the creator, require
  `hasPermission(rolePermissions,'leads.assign')` — mirror the PATCH gate; validate the target
  is an existing active lead-capable user; apply the identical guard to POST follow-ups.

#### [low] POST /api/crm/leads lets a sales_agent assign a new lead to an arbitrary user (near-duplicate of the above, leads-only) — `app/api/crm/leads/route.ts:194`
- **What's wrong:** Same defect scoped to the leads POST alone: `assignedTo` is taken from the
  body with no `leads.assign` check and no target validation, while PATCH gates it. Blast radius
  is limited to records the agent itself introduces (no access to others' existing leads).
- **Failure scenario:** `sayed` POSTs `assigned_to:'ghost_user'` → the lead is scoped to nobody
  and hidden from every non-admin until an admin intervenes; or `assigned_to:'elharm'` plants a
  lead in another agent's pipeline.
- **Suggested fix:** Mirror the PATCH gate; else force `assignedTo = auth.pyraUser.username`.
  (Fixing the medium finding above resolves this.)

#### [low] convert-to-customer & portal-access gate only on `leads.manage` with no `canAccessLead` scope — `app/api/crm/leads/[id]/convert-to-customer/route.ts:75`
- **What's wrong:** Both routes authorize with `requireApiPermission('leads.manage')` then perform
  cross-entity mutations (create client + Auth user; flip `portal_active`) with **no** ownership
  scope check. `leads.manage` is a normal grantable permission (member of `ALLOWED_EXTRA_PERMISSIONS`),
  and `canAccessLead`/`getLeadScopeFilter` only widen scope for `role==='admin'` — ignoring the
  `leads.manage` "see all leads" semantics documented at `rbac.ts:211`.
- **Failure scenario:** An admin grants `extra_permissions:['leads.manage']` to `sayed`; he can now
  convert or portal-enable any other agent's closed_won lead cross-scope, yet cannot even view those
  same leads in the scoped list/detail endpoints — an inconsistent partly-admin surface.
- **Suggested fix:** Add `canAccessLead(...)` to both routes (as every other per-lead mutation does),
  OR restrict them to a true admin-only check; separately make `getLeadScopeFilter`/`canAccessLead`
  honor `hasPermission(perms,'leads.manage')`.

#### [low] Attachments list/upload responses leak internal `storage_path` — `app/api/crm/leads/[id]/attachments/route.ts:157`
- **What's wrong:** GET selects `storage_path` and returns the full row via `{ ...(row as
  PyraLeadAttachment), public_url }`; POST likewise spreads the inserted row (line 449). These are
  client-PII files in the PRIVATE `pyra-private` bucket, and the sibling HR Documents Vault locks the
  invariant "storage_path NEVER leaves the server." The client only consumes `public_url`.
- **Failure scenario:** A sales_agent opens the مرفقات tab; the GET response exposes
  `lead-attachments/{lead_id}/{ts}-{nanoid}.jpg` for every row, disclosing the private bucket's
  internal path/naming scheme to the browser.
- **Suggested fix:** Destructure and drop `storage_path` before returning (mirror the HR vault):
  `const { storage_path, ...rest } = row; return { ...rest, public_url };`.

*(See "Needs manual confirmation" for the `team-performance` scoping finding.)*

### Bugs & Correctness

#### [medium] convert-to-customer crashes (500) for any lead with a NULL company — `app/api/crm/leads/[id]/convert-to-customer/route.ts:186`
- **What's wrong:** The `pyra_clients` insert passes `company: lead.company` verbatim.
  `pyra_sales_leads.company` is `text NULL` (leads only require name+phone), but
  `pyra_clients.company` is `varchar NOT NULL`. Inserting an explicit NULL into a NOT NULL column
  raises a constraint violation → `insertError` truthy → `apiError('فشل إنشاء سجل العميل', 500)`
  with no fallback and no hint of the real cause.
- **Failure scenario:** A B2C lead (name+phone, no company) is approved to closed_won; admin clicks
  "تحويل لعميل دائم" → 500. Any created Auth user is rolled back, but conversion is permanently
  impossible until someone backfills a company. The core won→customer journey is blocked for a whole
  class of leads.
- **Suggested fix:** `company: lead.company || lead.name || contactName`; optionally validate earlier
  and return a clear `apiValidationError`.

#### [medium] Customer dossier LTV/payment-health/revenue KPIs count only contract-linked invoices — `app/api/crm/customers/[lead_id]/dossier/route.ts:227`
- **What's wrong:** Invoices are fetched only via `.in('contract_id', contractIds)`; LTV and the
  payment health factor therefore only see contract-attached invoices. Any standalone invoice
  (`client_id` set, `contract_id = null` — the default `/api/clients`→invoice path) is ignored.
  The health-score spec defines the payment factor with no contract qualifier, and LTV is "lifetime
  value".
- **Failure scenario:** A converted customer with three paid standalone invoices totalling 60,000 AED
  but no contract shows `ltv=0`, `payment=0`, `recent_invoices_paid_on_time_pct=0` — a healthy paying
  client rendered as zero-revenue / at-risk.
- **Suggested fix:** When `lead.client_id` is set, also fetch `.eq('client_id', lead.client_id)` and
  union (de-dupe by id) with the contract-linked set.

#### [medium] Dashboard KPIs and funnel sum `expected_value` across mixed currencies but label it AED — `app/api/crm/dashboard/kpis/route.ts:63`
- **What's wrong:** `pyra_sales_leads.expected_value_currency` exists but is never fetched; `kpis`
  sums `expected_value` currency-blind for `pipeline_value`, `avg_deal_size`, and `forecast_close_value`,
  and sums `retainer_amount` ignoring contract currency for MRR. Responses are hard-labeled `total_aed`
  and the UI renders `formatCurrency(..., 'AED')`. `funnel/route.ts:41` has the identical defect.
- **Failure scenario:** One EGP lead (100,000) + one AED lead (50,000) → the KPI card shows
  `pipeline_value = 150,000 AED`, inflated by ~94k of misrepresented EGP; avg/forecast/funnel are
  similarly corrupted.
- **Suggested fix:** Group/return totals per currency (as HR payroll does) and render one figure per
  currency, or convert via a rate table — at minimum stop labeling a mixed-currency sum `total_aed`.

#### [medium] deals-at-risk & ai-insights drop active leads whose `is_converted` is NULL — `app/api/crm/dashboard/deals-at-risk/route.ts:39`
- **What's wrong:** `is_converted` is nullable; both routes filter `.eq('is_converted', false)`,
  which compiles to `WHERE is_converted = false` and excludes NULL rows (`NULL = false` → NULL).
  Migrations 010/011 explicitly treat `is_converted IS NULL` as not-converted and leave legacy leads
  in active stages with NULL, while `kpis` filters by stage only — so the surfaces disagree.
- **Failure scenario:** A legacy/imported lead in `stg_negotiation` with `is_converted=NULL`, idle 30
  days, is silently excluded from both the "deals at risk" list and the `idle_warning` insight; the
  agent is never prompted and the stale deal is invisible.
- **Suggested fix:** Use `.or('is_converted.is.null,is_converted.eq.false')` (or
  `.not('is_converted','is',true)`), matching migrations 010/011 and the stage-only `kpis` logic.

#### [low] move-stage `contract_signed` accepts any existing contract/invoice id as the signed attachment — `app/api/crm/leads/[id]/move-stage/route.ts:159`
- **What's wrong:** The attachment guard only checks the contract/invoice **exists** (`.eq('id',
  attachment.id)`), never that it belongs to this lead or its client. An agent who legitimately owns
  the lead can pass any contract/invoice id in the system as the "signed proof".
- **Failure scenario:** An agent moves their lead to "تم توقيع العقد" supplying another client's
  invoice id; the closed_won_pending activity + manager notification are created with an unrelated
  attachment label; a manager may approve a deal with no real signed contract.
- **Suggested fix:** Scope the attachment lookup — contracts require `lead_id = leadId OR client_id =
  lead.client_id`; invoices require `client_id = lead.client_id`.

#### [low] Activities GET passes the `before` cursor to `.lt()` without validation → 500 on malformed input — `app/api/crm/leads/[id]/activities/route.ts:57`
- **What's wrong:** `before` is read as a raw trimmed string and passed straight to
  `.lt('created_at', before)` against a timestamptz column. A non-timestamp value raises Postgres
  22007 → `apiServerError()` (500) instead of a 400/validation error.
- **Failure scenario:** `GET .../activities?before=abc` returns 500 for what is actually bad client
  input, plus log noise.
- **Suggested fix:** Parse `new Date(before)` and return `apiValidationError` on `NaN` before feeding
  it to `.lt()`.

#### [low] Dossier LTV/MRR sum across currencies but hardcode `currency:'AED'` — `app/api/crm/customers/[lead_id]/dossier/route.ts:436`
- **What's wrong:** `kpis.ltv` and `kpis.mrr` are plain numeric sums over contracts/invoices whose
  `currency` can differ, then the block hardcodes `currency:'AED'`. Violates the workspace's "never sum
  across currencies" rule.
- **Failure scenario:** An AED retainer + an EGP project → `ltv` sums EGP and AED as one number tagged
  AED; MRR inflates if any retainer is non-AED.
- **Suggested fix:** Group the KPI sums by currency and return per-currency totals, or surface the
  dominant/contract currency instead of a hardcoded literal.

#### [low] Health-score recency factor cannot produce the 30-90d bucket from activity data — `app/api/crm/customers/[lead_id]/dossier/route.ts:187`
- **What's wrong:** Activities are fetched with `.gte('created_at', thirtyDaysAgo)` and reused for BOTH
  engagement AND recency. Since `activities[0]` can never be >30 days old, the `days<90` (→10) and else
  (→0) recency branches are unreachable from the activity path; they only fire via the `last_contact_at`
  fallback (null-only when there are zero recent activities).
- **Failure scenario:** Latest activity 45 days ago + `last_contact_at=null` → `recencyScore=0` where the
  spec expects 10 (30-90d), under-scoring the customer by 10 points and possibly flipping the color band.
- **Suggested fix:** Fetch the single most-recent activity for recency with no 30-day floor
  (`order('created_at',desc).limit(1)`), keeping the windowed query only for engagement.

#### [low] POST follow-ups: `due_at` not validated when `reminder_at` is supplied → Postgres error leaks as 500 — `app/api/crm/follow-ups/route.ts:152`
- **What's wrong:** `due_at` ISO validity is checked only inside the ELSE branch that runs when
  `reminder_at` is absent. When a valid `reminder_at` is supplied, `due_at` is never validated before
  insert into the `NOT NULL timestamptz` column → DB rejects it → `apiServerError` with the raw Postgres
  message.
- **Failure scenario:** `{ due_at:'tomorrow 3pm', reminder_at:'2026-07-03T10:00:00Z' }` passes app
  validation, Postgres rejects `'tomorrow 3pm'`, API returns 500 with a leaked DB error string instead of
  a 422.
- **Suggested fix:** Validate `dueAt` once, unconditionally, right after the required-field check.

#### [low] ai-insights "follow-ups due today" uses server-local day, not Dubai day — `app/api/crm/dashboard/ai-insights/route.ts:158`
- **What's wrong:** Rule 4 builds the window with `new Date()` + `setHours(0,0,0,0)`/`setHours(23,59,59,999)`,
  which on the UTC Coolify server is the UTC calendar day, not Asia/Dubai. Violates the Phase 15.1
  `dubaiDayKey()` lock.
- **Failure scenario:** At 01:30 Dubai (21:30 UTC prior day) the window covers the prior UTC day; a
  follow-up due 10:00 Dubai next day is missed while a prior-day one is wrongly counted — mis-bucketing
  every boundary-window follow-up.
- **Suggested fix:** Derive the window from `dubaiDayKey(new Date())` / `toDubaiIso` (UTC+4).

#### [low] ai-insights `idle_warning` ignores `last_contact_at`, contradicting its own comment & diverging from deals-at-risk — `app/api/crm/dashboard/ai-insights/route.ts:78`
- **What's wrong:** The Rule 1 comment says "last_contact_at older than 7d (or null)" but the code
  determines idleness solely from `pyra_lead_activities` in the last 7 days, never reading
  `last_contact_at`. `deals-at-risk:63` uses the greatest of `last_contact_at` and latest activity, so the
  two surfaces (both linking to `at_risk`) can disagree and the insight over-counts.
- **Failure scenario:** An agent phones a lead today (`last_contact_at` set) but logs no activity row →
  ai-insights counts it idle and raises a "high" stale-deal banner, while deals-at-risk correctly omits it.
- **Suggested fix:** Include `last_contact_at` in the idle determination (idle only when both it and the
  latest activity are older than the threshold, or null).

#### [low] AI-insights `followups_today` server-local day boundaries (second instance) — `app/api/crm/dashboard/ai-insights/route.ts:158`
- **What's wrong:** Same Dubai-day defect as above, re-flagged from the correctness-sweep slice — Rule 4's
  `setHours`-derived window is server-local (UTC) rather than Dubai. Duplicate of the ai-insights
  "follow-ups due today" finding; listed for traceability.
- **Failure scenario:** Follow-up due 02:00 Dubai (22:00 UTC prior day) counted as "due today" a day early;
  late-Dubai-day follow-ups excluded — off-by-one near midnight.
- **Suggested fix:** Reuse `dubaiDayKey()`/`toDubaiIso`. (One fix resolves both entries.)

#### [low] Legacy lead GET/PATCH: recomputed score never persisted (`void` update) — `app/api/dashboard/sales/leads/[id]/route.ts:78`
- **What's wrong:** Both GET (78) and PATCH (176) recompute the score and try to persist via `void
  supabase.from('pyra_sales_leads').update({ score }).eq('id', id)` — a lazy-thenable never dispatched.
  The stored `score` is only ever written at POST creation. The detail response returns the live score,
  the list/kanban reads the stale stored column.
- **Failure scenario:** A lead accrues activity after creation; detail shows an updated score (78) while
  the pipeline list keeps the creation-time score (40). (Impact is negligible today — these legacy
  routes/hooks are unwired dead code post-Phase-12.)
- **Suggested fix:** Await the score update (or fold it into the PATCH's own UPDATE).

#### [low] Non-404 dossier errors leave the Active Customer Page stuck in a permanent skeleton — `app/dashboard/crm/customers/[id]/customer-detail-client.tsx:51`
- **What's wrong:** Only the 404 error case is handled. For any other error (500, exhausted retries),
  `error` is set, `isLoading=false`, `dossier=undefined`, and the render falls through to permanent
  skeletons + the literal `جاري تحميل بيانات العميل...` with no error message and no retry.
- **Failure scenario:** The dossier endpoint 500s; after 3 retries the user is left on a half-rendered
  page (header skeleton, "—" stat cards, spinning health card, perpetual loading text) with no feedback.
  (Partial self-heal via `refetchOnWindowFocus:true`.)
- **Suggested fix:** Add a generic error branch after the 404 check rendering an `EmptyState` with a retry
  button.

### Data Integrity

#### [medium] Reopening a closed_won lead leaves `is_converted`/`converted_at`/`win_probability` stale — `app/api/crm/leads/[id]/move-stage/route.ts:189`
- **What's wrong:** The reopen branch builds `updates` with only `stage_id`, `updated_at`, and
  conditionally `win_probability`. It never resets `is_converted`, `converted_at`, or
  `win_probability_overridden`. Because approve sets `win_probability_overridden=true`, the guard
  `if (!leadBefore.win_probability_overridden)` is false on reopen, so `win_probability` stays 100 and the
  override flag is stuck true forever (disabling recompute on all future moves).
- **Failure scenario:** Admin reopens a fallen-through deal to negotiation → the lead shows
  `stage=negotiation` but `win_probability=100`, `is_converted=true`, `converted_at` set. It still appears
  in `/dashboard/crm/customers` as a converted customer and every later stage move keeps the probability
  frozen at 100.
- **Suggested fix:** In the reopen branch reset `is_converted=false`, `converted_at=null`,
  `win_probability_overridden=false`, `win_probability=STAGE_DEFAULT_WIN_PROBABILITY[toStage]`.

#### [medium] convert-to-customer: client row created but lead-link failure leaves an unrecoverable stuck state — `app/api/crm/leads/[id]/convert-to-customer/route.ts:220`
- **What's wrong:** Multi-write with no transaction and only partial rollback. If the `pyra_clients`
  insert succeeds but the `UPDATE pyra_sales_leads.client_id` fails, the code only logs a warning and
  still returns `apiSuccess({ created:true })`. On retry, `lead.client_id` is still null so the idempotency
  short-circuit is skipped, the duplicate-email check finds the just-created client, and returns 409. The
  in-code comment claiming idempotency will recover is provably false.
- **Failure scenario:** A transient failure on the lead UPDATE; client + Auth user exist, lead stays
  unlinked, response is 200 created:true. Admin retries → 409 "email already registered". The lead is
  permanently unlinkable via the UI.
- **Suggested fix:** Treat `linkError` as fatal (roll back the client row + Auth user and error out), OR
  make the duplicate path recover by re-linking the existing client and returning `created:false`.

#### [medium] Legacy lead-create: timeline activity + audit-log rows never persist (`void` lazy-thenable) — `app/api/dashboard/sales/leads/route.ts:228`
- **What's wrong:** Both fire-and-forget writes use `void supabase.from(...).insert({...})` with no
  `.then()`/await (line 228 → `pyra_lead_activities`; line 238 → `pyra_activity_log`). A PostgrestBuilder
  only dispatches on `.then()`, so both inserts are built but never sent — the exact bug already fixed in
  the sibling bulk route.
- **Failure scenario:** An agent creates a lead via the WhatsApp chat dialog; the lead row is inserted but
  the Activity-tab entry and the audit-log row are never written — a silent audit gap and dual-write drift
  vs the CRM create path.
- **Suggested fix:** Await the inserts (log errors), and route the audit row through `logActivity()`.

#### [medium] Legacy follow-up create: `lead.next_follow_up` is never updated (`void` update) — `app/api/dashboard/sales/follow-ups/route.ts:93`
- **What's wrong:** After inserting the follow-up, `void supabase.from('pyra_sales_leads').update({
  next_follow_up, updated_at }).eq('id', lead_id)` is never dispatched, so the parent lead's
  `next_follow_up` never reflects the scheduled follow-up. The parallel CRM route correctly chains
  `.then()`.
- **Failure scenario:** An agent schedules a follow-up from chat; the follow-up row is created but
  `next_follow_up` stays null/stale, so the contact-panel "متابعة قادمة" indicator and any pipeline sort
  show wrong/missing data.
- **Suggested fix:** `if (lead_id) { const { error } = await supabase...update(...).eq('id', lead_id); if
  (error) console.error(...) }`.

#### [low] Attachment DELETE removes the storage object BEFORE the DB row, inverting the documented failure priority — `app/api/crm/leads/[id]/attachments/[attachmentId]/route.ts:101`
- **What's wrong:** The handler removes the storage object first (best-effort, log-and-continue) and only
  then deletes the DB row (returns 500 on failure). If the storage remove succeeds but the DB delete fails
  transiently, the row survives while its object is gone — the opposite of the route's own documented
  intent. The sibling POST uses the correct DB-first ordering.
- **Failure scenario:** Storage.remove succeeds, the following `.delete()` errors transiently → 500, row
  remains → every later GET signs a URL for a missing object → a permanently broken image/audio cell.
- **Suggested fix:** Reorder — DB DELETE first (fail on error), then best-effort storage remove afterward.

#### [low] Legacy lead-create: in-app "lead assigned" notification never delivered (`void` insert) — `app/api/dashboard/sales/leads/route.ts:251`
- **What's wrong:** When a lead is assigned to someone else, the bell notification is written via `void
  supabase.from('pyra_notifications').insert({...})` — never dispatched. The email path (`notifyLeadAssigned`,
  a real async fn) DOES fire, so the assignee gets the email but no in-app notification — an inconsistent
  half-delivery that also bypasses the mandated `notify()` helper.
- **Failure scenario:** Admin creates a lead assigned to another agent; the agent receives the assignment
  email but no bell notification ever appears.
- **Suggested fix:** Replace with `await notify(supabase, { to, type:'lead_assigned', ... })`.

#### [low] Legacy lead PATCH: stage-change activity never logged (`void` insert) — `app/api/dashboard/sales/leads/[id]/route.ts:139`
- **What's wrong:** On a stage change the PATCH records the activity with `void supabase.from(
  'pyra_lead_activities').insert({...})` — never dispatched, so the `stage_change` timeline entry is never
  written. The CRM move-stage path logs it correctly, creating dual-write drift. (This legacy PATCH is
  currently unwired in the UI.)
- **Failure scenario:** A lead moved via the legacy update path updates its stage but records no
  `stage_change` activity, leaving pipeline-movement audit history incomplete.
- **Suggested fix:** Await the insert and log errors.

#### [low] Legacy convert-to-client route inserts a non-existent `notes` column (+ omits NOT NULL fields) → conversion always fails — `app/api/dashboard/sales/leads/[id]/convert/route.ts:54`
- **What's wrong:** The `pyra_clients` insert sends `notes: lead.notes || null`, but the table has no
  `notes` column (PGRST204 rejects it). It also omits NOT NULL `password_hash`/`portal_active` and passes a
  nullable `email`. There is no try/catch, and line 129 computes an unused `phone`. The working CRM route
  sets all required fields correctly.
- **Failure scenario:** Any caller (old bookmark, automation) POSTs to this route → insert fails ("Could
  not find the notes column…" / NOT NULL violation) → 500, no client created. (Endpoint is currently
  orphaned — nothing calls it — and fails safe with no partial writes.)
- **Suggested fix:** Remove `notes`; add `password_hash:''`, `portal_active`, `role:'client'`, `is_active`,
  and a dup-email guard mirroring the CRM route; wrap in try/catch; better, deprecate this endpoint.

#### [low] Lead Deals tab hardcodes 'AED' for invoice amounts in a multi-currency system — `components/crm/lead-detail/lead-deals-tab.tsx:199`
- **What's wrong:** Invoice totals render with `formatCurrency(Number(inv.total) || 0, 'AED')` — hardcoded
  currency. Contracts and quotes on the same card pass their real currency, and the invoices header prints
  the true `payments_summary.currency`. The dossier route never selects `currency` and `LeadDetail.invoices`
  omits it, so the UI cannot show the real value.
- **Failure scenario:** An EGP invoice for 25,000 EGP renders as "AED 25,000.00" beneath a correct
  "مدفوع EGP …" header — wrong symbol on the same card.
- **Suggested fix:** Add `currency` to the invoices SELECT + the `LeadDetail.invoices` type, and render
  `formatCurrency(Number(inv.total) || 0, inv.currency || 'AED')`.

### Crons

#### [medium] Idle-check activity-recency SELECT error is silently swallowed → false "idle" warnings on transient DB failure — `app/api/cron/lead-idle-check/route.ts:114`
- **What's wrong:** The "most-recent activity per lead" query destructures only `{ data: actData }` and
  never inspects the error (unlike the primary leads SELECT which checks `leadErr` and returns 500). On a
  transient failure `actData` is null, `lastActivityByLead` stays empty, and the idle filter evaluates each
  lead using only `last_contact_at` (activity recency treated as never-touched).
- **Failure scenario:** The activities SELECT transiently errors; a lead with a fresh note (activity row)
  but null `last_contact_at` is misclassified idle → false `idle_warning` row + false "1 deal idle 7+ days"
  agent notification, repeated for every such lead, and the stray warning poisons the 7-day dedup.
- **Suggested fix:** Capture and check the error; on failure `logError` + `apiServerError()` (mirror the
  leads-SELECT guard) instead of producing false positives.

#### [low] 7-day idle dedup & per-agent daily notif-dedup SELECT errors swallowed → duplicate warnings/notifications — `app/api/cron/lead-idle-check/route.ts:156`
- **What's wrong:** The Q-11-2 dedup query (156) and the Q-11-3 notif-dedup query (235) both destructure
  only `{ data }`. If either read fails, the dedup set is empty, so already-warned leads are re-warned and
  already-notified agents are re-notified — defeating the documented idempotency guarantees.
- **Failure scenario:** Lead L was flagged idle yesterday; today the dedup SELECT transiently errors → L
  treated as newly-idle → a second `idle_warning` row within the 7-day window; likewise an agent gets a
  duplicate daily summary.
- **Suggested fix:** Check the error on both reads; fail-closed (`apiServerError`) or at least `logError`.

#### [low] Leads with NULL `stage_id` are silently excluded from the idle check (NOT IN NULL semantics) — `app/api/cron/lead-idle-check/route.ts:82`
- **What's wrong:** `.not('stage_id','in',(...))` compiles to `NOT (stage_id IN (...))`; for `stage_id IS
  NULL` this is NULL (not TRUE), so PostgREST excludes the row. `stage_id` is nullable, so any assigned,
  non-converted, stage-less lead is never evaluated for idleness.
- **Failure scenario:** A bulk-imported lead with `assigned_to` set, `is_converted=false`, `stage_id=NULL`
  sits untouched 30 days and never triggers an idle warning. (Normal create paths always assign a stage, so
  exposure is limited to out-of-band data.)
- **Suggested fix:** Include NULL-stage leads, e.g. `.or('stage_id.is.null,stage_id.not.in.(...)')`.

#### [low] `agent_idle_summary` entity_id uses UTC `.slice(0,10)` → off-by-one Dubai-day label — `app/api/cron/lead-idle-check/route.ts:265`
- **What's wrong:** `entity.id = dubaiTodayUtcIso.slice(0,10)`, where `dubaiTodayUtcIso` is the UTC instant
  of Dubai-midnight (always `(D-1)T20:00:00Z`), so `.slice(0,10)` yields the previous UTC date. Currently
  cosmetic (real dedup is the `created_at >= dubaiTodayUtcIso` query), but a wrong Dubai-day label.
- **Failure scenario:** On Dubai date 2026-07-02 the cron stamps `entity_id='2026-07-01'`; any future
  consumer that groups/dedups by `(recipient, entity_id)` would mis-key it to the prior day.
- **Suggested fix:** `id: dubaiDayKey(new Date())` from `lib/utils/format.ts`.

#### [low] Lead-list `activity_count`/`last_activity_type` enrichment fetches activities with no row limit and counts in JS — `app/api/crm/leads/route.ts:113` *(UNCERTAIN — see below)*
- **What's wrong:** The enrichment query `.in('lead_id', ids).order('created_at',desc)` has no `.limit()`
  and no `head/count`; it materializes every activity row across the page of leads and tallies counts in JS.
  If the deployment configures PostgREST `db-max-rows`, the result silently truncates and counts become

- **Suggested fix:** Compute counts server-side (grouped count or batched head+count) and fetch only the
  single latest activity per lead; at minimum bound the query and document the assumption.

  `useUpdateLead` (and the sibling mutations for completeness).

#### [low] Owner filter is a one-way trap — dropdown collapses to the selected owner — `app/dashboard/crm/pipeline/pipeline-client.tsx:143`

#### [low] No keyboard-accessible way to change a lead's stage on the desktop pipeline board — `components/crm/pipeline/pipeline-board.tsx:84`

#### [low] Activity timeline day-dividers group by UTC day, not Dubai day — `components/crm/activity/activity-timeline.tsx:74`

- **Failure scenario:** A note logged at 02:00 Dubai on 2026-07-03 (= 2026-07-02T22:00Z) files under the

#### [low] Invoice rows display the raw internal ID instead of the human invoice number — `components/crm/lead-detail/lead-deals-tab.tsx:190`

#### [low] Activity filter chips & composer type chips lack toggle-state semantics for screen readers — `components/crm/activity/activity-timeline.tsx:133`

## Needs Manual Confirmation (UNCERTAIN verdicts)

These two findings describe accurate code observations but their impact is contingent on
deployment configuration or a not-yet-existing role, so they could not be reproduced as live
defects from code alone.

- **[low] Lead-list `activity_count` enrichment has no row limit — `app/api/crm/leads/route.ts:113`.**
  The unbounded `.in()` select + JS counting is real, but the truncation defect manifests only if the
  deployment sets PostgREST `db-max-rows` (default is unlimited; self-hosted Supabase doesn't set it) AND a
  single page collectively exceeds that cap. **Confirm whether `PGRST_DB_MAX_ROWS` is configured in
  production** before treating this as an active bug.

- **[low] `team-performance` applies no per-agent scoping — `app/api/crm/dashboard/team-performance/route.ts:25`.**
  The endpoint returns full-team figures with no direct-report restriction. This is correct for admin, and
  `crm_reports.team_view` is currently admin-only (intentionally omitted from sales_agent, no manager role
  exists). The exposure only materializes **if** an admin later grants `crm_reports.team_view` to a non-admin
  via `extra_permissions` or a new DB role. Treat as future-hardening: scope non-admins to
  `getDirectReports(username)` (mirroring ai-insights Rule 2) before that permission is ever delegated.
