# i18n Phase 6 — Admin Tail + Remaining Dashboard — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development, task-by-task. Execute sub-wave by sub-wave; each sub-wave ships + deploys independently (D4).

**Goal:** Make ALL remaining dashboard surfaces bilingual AR/EN — settings, admin/error-logs, users/roles/permissions/directory/org-chart + rbac display names, the 1131-string module-guide, clients/projects/files/teams, content-ops (pipeline/reviews/automations/integrations/storage/kb), and the misc pages + WhatsApp chat inbox — Arabic pixel-identical after every wave.

**Architecture:** next-intl WITHOUT locale routing (locale ← `pyra_locale` cookie ← DB). Deep-merge EN-over-AR fallback. Recipe per wave: new catalog namespace(s) → UI swap (`useTranslations`/`useStatusLabels`) → API swap (`getTranslations('api')`) → manifest append → QA both locales → commit → push → Coolify deploy.

**Tech stack:** next-intl 4.x, `messages/{ar,en}/*.json`, `lib/i18n/status-labels.ts`, `scripts/i18n-check.ts` guard.

**Scale (6 censuses in `.superpowers/sdd/p6-discovery/`):** ~4,700 translatable strings across ~300 files — the largest phase. Split into 5 independently-deployable sub-waves.

## Global Constraints (inherit ALL Phase 0–5 locks; Phase-6-specific below)

- **AR byte-identical** (verbatim extraction); EN authored neutral/formal. Money locale-invariant (`formatCurrency` no locale param; `toLocaleString(numberLocale)` `ar-AE`↔`en-AE`; dates via `formatDate(x,fmt?,locale)` or in-place `'ar-XX'↔'en-XX'` preserving AR digits — NEVER reroute a `toLocaleDateString('ar-*')` to date-fns `formatDate` default, it changes digits/format).
- **NO JSON arrays in catalogs** — `deepMerge` replaces arrays wholesale (breaks per-item AR fallback). module-guide `tips[]`/`keywords[]` (6b) and rbac `PERMISSION_MODULES` array (6a) become index-keyed objects; the ASCII index array stays in CODE (nav-config.ts precedent).
- **Same-line `// i18n-exempt:` tags** — scanner is LINE-SCOPED (recurred P4-T4/T6, P5-T3/T9). Applies to code comments too.
- **Shadow-map RECONCILIATION (sanctioned convergence, document each):** many local `*_LABELS` maps duplicate existing statuses entities with CONFLICTING wording (contract «عقد»/«متعاقد» vs canonical «عقد مؤقت»; client «معطل» vs «غير نشط»; source enum ±`import`/`accepted` keys). Migration wires `useStatusLabels(entity)` + DELETES the shadow map, converging to the canonical statuses.json wording — call out every text change in the task report (P4/P5 drift-fix doctrine).
- **New status entities to ADD (T1 of the owning wave):** `projectStatus` (6c — none exists; 3 shadow maps + server validStatuses), `errorSeverity`/`errorEnvironment`/`resolvedState` (6a error-logs — reconcile the 3× redundant maps + gender mismatch «محلولة»/«محلول»), plus any the census flags (webhookEvent for 6d, fileCategory for 6c — but fileCategory has a PERSISTED-data copy in upload-complete route: migrate the UI label, leave the auto-tag DB write Arabic + exempt).
- **Bilingual DB rows NEVER migrated** (`pyra_roles.name_ar`, `pyra_business_entities.name_ar`/`name_en`, doc/eval/pipeline `name_ar`): render `locale==='ar' ? name_ar : (name||name_ar)`; the DB value stays.
- **Shared dashboard+portal components** (`components/files/*` file-preview/explorer/context-menu/hover-preview; possibly others): migrate the Arabic NOW in 6c using `useTranslations` (works in both shells — both mount `NextIntlClientProvider`). **Flag each in the report so Phase 7 does NOT double-migrate.**
- **Non-translatable content = exempt** (board-template precedent): the `whatsapp/ai-suggest` rule-based Arabic NLU engine + LLM prompts (~93 lines), KB freeform article prose, seeded DB content. Same-line exempt tags.
- **Notifications page (6e):** migrate page CHROME (heading, filters, `NOTIF_TYPES` map, empty state); the stored `n.title`/`n.message`/`n.source_display_name` are DB values rendered as-is → Phase 8, leave.
- **Dev-internal copy leaks (Phase 13 lock):** `error-logs-client.tsx:120` "Phase 14.1" in user copy → replace with user-facing text (or "قريباً" idiom). No "Phase X"/"TODO"/"قيد البناء" in shipped UI.
- **ACTION_LABELS activity dictionary** duplicated ≥4× (activity-client, api/activity/export, client-detail, file-preview) WITH drift (export missing HR/payroll keys) → ONE catalog source (`activity.actions.*` or reuse existing), reconcile the drift, wire all consumers. (Board's copy already migrated Phase 2 — check for a reusable key set first.)
- **Hands-off (do NOT fix in this phase — avoid conflicts / out of scope):** raw `fetch()`/`useState` data layers (permissions/reviews/integrations/activity/login-history/reports/sales-approvals clients — string-externalize ONLY, backlog the React Query migration); >300-line file splits (unless a task brief says so); the WhatsApp `webhook.ts` business logic.
- **Cron/webhook/external routes stay Arabic** (machine consumers). Manifest appended per wave (verify each path exists — fail-loud); EXCLUDE nothing already migrated.
